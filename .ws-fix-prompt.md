## El bug

El endpoint WebSocket en `src/api/routes/ws-dashboard.ts` está registrado como `fastify.get('/ws', { websocket: true }, ...)` dentro del scope `/api/v1`. El cliente del dashboard se conecta a `wss://core.fomo.com.ar/api/v1/ws?projectId=X` pero recibe `HTTP/1.1 401 Unauthorized` en el handshake. Debería recibir `HTTP/1.1 101 Switching Protocols`.

El middleware en `src/api/auth-middleware.ts` tiene una exención que debería aplicar pero no lo está haciendo:

```typescript
const WS_PATH = '/api/v1/ws';
if (pathname.startsWith(WEBHOOK_PREFIX) || pathname === WS_PATH || pathname.startsWith(`${WS_PATH}?`)) {
  return;
}
```

Dato clave: el 401 NO genera ningún log — ningún middleware loguea "Rejected" ni "Missing Authorization". Algo rechaza antes de loguear.

### PASO G — Diagnóstico según logs

Con el output de los logs, identificá cuál de estos 4 escenarios aplica:

**Escenario 1**: Aparece `[WS-DEBUG] auth-middleware hit` con `exempt_ws: false` y `pathname` distinto a `/api/v1/ws`.
→ **Fix**: Ajustá la comparación en `auth-middleware.ts` para matchear la realidad. Por ejemplo si `pathname` es `/ws` (sin prefijo), cambiá el check a `pathname.endsWith('/ws')`.

**Escenario 2**: Aparece `[WS-DEBUG] auth-middleware hit` con `exempt_ws: true` (o sea exención OK) pero igual da 401.
→ **Fix**: El problema está en `requireProjectAccess` o `requireScope` que también corren como hooks después del auth middleware y NO tienen exención para WS. Agregales la misma exención al principio de cada uno:
```typescript
const pathname = request.url.split('?')[0] ?? '';
if (pathname === '/api/v1/ws' || pathname.startsWith('/api/v1/ws?') || pathname.endsWith('/ws')) return;
```

**Escenario 3**: NO aparece NINGÚN log `[WS-DEBUG]` para el request del WebSocket.
→ **Fix**: El hook `onRequest` no se ejecuta para requests con `Upgrade: websocket`. Esto puede pasar si `@fastify/websocket` intercepta el upgrade antes de que los hooks del scope `/api/v1` se ejecuten. Investigá la documentación de `@fastify/websocket` y ajustá cómo se registra el auth — posiblemente hay que usar un `preHandler` específico del route del WS o usar la opción `preHandler` del plugin.

**Escenario 4**: Aparece un log distinto que menciona `admin-auth` rechazando el WS.
→ **Fix**: El hook de admin-auth está aplicándose donde no debería. Revisá dónde se registra y agregále exención de WS.
