## ITEM 2 — Backups automáticos de Postgres

**Contexto**: No hay backups automatizados. El último manual es `/root/nexus_core_pre_schema_fix.dump` (613K, 20 abril 2026).

**Lo que hay que hacer**:

1. Crear el script `/root/backup-nexus.sh` en el VPS con este contenido:

```bash
#!/bin/bash
# Backup de Postgres nexus_core — corre semanal vía cron
set -euo pipefail

BACKUP_DIR=/root/backups
LOG_FILE=$BACKUP_DIR/backup.log
TIMESTAMP=$(date +%Y%m%d_%H%M)
DUMP_FILE=$BACKUP_DIR/nexus_core_$TIMESTAMP.dump
CONTAINER=compose-generate-multi-byte-system-fqoeno-postgres-1
RETENTION_DAYS=28

mkdir -p $BACKUP_DIR

echo "[$(date -Iseconds)] Starting backup → $DUMP_FILE" >> $LOG_FILE

# Dump a /tmp del container, después copiar al host
if docker exec $CONTAINER pg_dump -U nexus -d nexus_core -F c -f /tmp/nexus_backup.dump 2>>$LOG_FILE; then
    if docker cp $CONTAINER:/tmp/nexus_backup.dump $DUMP_FILE 2>>$LOG_FILE; then
        SIZE=$(du -h $DUMP_FILE | cut -f1)
        echo "[$(date -Iseconds)] Backup OK ($SIZE) → $DUMP_FILE" >> $LOG_FILE
        docker exec $CONTAINER rm -f /tmp/nexus_backup.dump
    else
        echo "[$(date -Iseconds)] ERROR: docker cp failed" >> $LOG_FILE
        exit 1
    fi
else
    echo "[$(date -Iseconds)] ERROR: pg_dump failed" >> $LOG_FILE
    exit 1
fi

# Rotación: borrar backups más viejos de $RETENTION_DAYS días
find $BACKUP_DIR -name "nexus_core_*.dump" -type f -mtime +$RETENTION_DAYS -delete
echo "[$(date -Iseconds)] Rotation done (kept last ${RETENTION_DAYS}d)" >> $LOG_FILE
echo "" >> $LOG_FILE
```

Hacé el archivo ejecutable: `chmod +x /root/backup-nexus.sh`.

2. Probalo manualmente:

```bash
ssh hostinger-fomo "/root/backup-nexus.sh && tail -10 /root/backups/backup.log && ls -lh /root/backups/"
```

Esperado: log con "Backup OK" y el archivo `.dump` creado.

3. Instalá el cron — corre cada domingo a las 3 AM UTC:

```bash
ssh hostinger-fomo "(crontab -l 2>/dev/null; echo '0 3 * * 0 /root/backup-nexus.sh') | crontab -"
```

Verificá:

```bash

```

4. Documentá lo hecho en `.claude/skills/backups.md` (si no existe todavía, creala). El skill debe cubrir:
   - Qué hace el script y cada cuánto corre.
   - Dónde están los backups y cómo rotan.
   - Cómo correr un backup manual.
   - Cómo restaurar un backup sobre DB actual (referenciar el playbook que ya existe en memoria).
   - Cómo ver el log.

5. Commit del skill:


## ITEM 3 — Investigar y cerrar el 404 de /cost

**Contexto**: El dashboard pega a `/projects/<projectId>/cost` y recibe 404. No afecta chat/agentes pero ensucia la consola del browser. Los logs del browser muestran:

```
GET https://fomo-core-dashboard.fomo.com.ar/projects/<id>/cost?_rsc=xxx 404 (Not Found)
```

Eso es un Next.js RSC request del dashboard (request al propio Next, no al backend). Así que el problema puede estar en el dashboard también, no solo en el backend.

**Lo que hay que hacer**:

1. Verificar cómo está registrada la ruta cost en el backend:

```bash

grep -r "costRoutes\|cost.ts" src/api/routes/index.ts
```

Según lo que sabemos, en `src/api/routes/index.ts` está:
```ts
await fastify.register(
  (f) => { costRoutes(f, deps); },
  { prefix: '/cost' },

```

Eso crearía endpoints bajo `/api/v1/cost/...`, no bajo `/api/v1/projects/:projectId/cost`. Posible mismatch: el dashboard pega a `/projects/.../cost` pero el backend lo expone en `/cost/...`.

2. Verificar cómo pega el dashboard. El submódulo vive en `dashboard/`. Buscá:

```bash

grep -rn "/cost" src/ --include="*.ts" --include="*.tsx" | head -20
```

Identificá qué URL arma el dashboard para pedir cost data.

3. Determinar la naturaleza del 404:
   - ¿Es una página de Next.js que no existe? (el URL `/projects/<id>/cost` sería una ruta de Next app router)
   - ¿O es un fetch al backend que también devuelve 404?

4. Según el diagnóstico, aplicar fix apropiado. Tres posibilidades:
   a. El dashboard tiene una ruta `/projects/[id]/cost` que no existe como page. Crear la page o eliminar el link.
   b. El backend debería exponer `/projects/:id/cost` pero no lo hace. Registrar la ruta.
   c. El dashboard apunta mal. Corregir el URL.

5. Si el fix es solo del dashboard, commitea ahí (es submódulo, repo aparte). Si es del backend, pre-push checklist + deploy al repo principal.

**IMPORTANTE**: si el 404 viene de una página de Next.js que simplemente no existe (escenario `a`), el fix más conservador es **eliminar el link del sidebar/navegación** que intenta ir a esa página. No creas páginas nuevas que no existían en el diseño. Si no sabés si la página debería existir, dejá el item como "PARCIAL — requiere decisión de producto" y reportá para hablarlo conmigo.
